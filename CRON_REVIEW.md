# Lifestyle Cron Review

This file is the source-controlled snapshot of the current OpenClaw cron surface, why each cron exists, and what has already been improved in the repo-managed prompts.

## Current Cron Jobs

All schedules below are in `Asia/Jerusalem`.

### Every day

1. `07:15` — `Lifestyle morning plan`
   - User-facing Telegram plan for today.
   - Reads fresh task state, calendar context, and OpenViking-backed scheduling suggestions.
   - Improvement already applied: prefer `preferred_slots` from `scheduling-decisions` before generic fallback slots.

2. `07:45` — `Lifestyle scheduling sweep`
   - Silent hard-time scheduling pass.
   - Purpose is different from the `07:15` cron: this one auto-places only safe explicit hard-time work, without sending a planning message unless something real changed or failed.
   - Improvement already applied: it does not auto-place flexible work.

3. `21:30` — `Daily overview with OpenClaw`
   - User-facing evening summary.
   - Runs `close-day` first, then sends the exact `evening-summary` output.
   - Improvement already applied: message formatting is now generated by the wrapper instead of freeform agent formatting.

4. `21:35` — `Lifestyle daily completion poll`
   - Silent cron that sends Telegram-native multi-select completion polls for confirmable `today` tasks.
   - Improvement already applied: excludes `@auto-done` and `@no-check` tasks so the poll only asks what actually needs confirmation.

5. `every 1 minute` — `Lifestyle daily completion poll watcher`
   - Silent watcher that checks whether a sent nightly poll has received its first answer, closes it after the delay window, and applies results.
   - Improvement already applied: answer ingestion is now persisted and machine-readable, not prompt-only.

6. `every 1 minute` — `Lifestyle telegram poll reply watcher`
   - Silent watcher for generic wrapper polls sent with `--notify-on-answer true`.
   - Improvement already applied: a voted poll can now trigger an automatic follow-up message with the selected options.

7. `every 30 minutes` — `Lifestyle reconcile`
   - Silent lifecycle cleanup and consistency pass.
   - Improvement already applied: respects `@auto-done` / `@no-check`, rolls recurring work forward, and updates miss counts.

### Weekly

8. `Saturday 09:00` — `Weekly overview with OpenClaw`
   - User-facing weekly review and next-week planning message.
   - Improvement already applied: `Schedule these` only offers actually missing future coverage and prefers OpenViking-backed timing suggestions.

9. `Saturday 09:20` — `Lifestyle weekly scheduling sweep`
   - Silent weekly hard-time placement pass that runs after the weekly overview.
   - Improvement already applied: only explicit hard-time work is auto-placed.

10. `Sunday 20:00` — `Life priority meeting with OpenClaw`
   - User-facing weekly priority check-in.
   - Improvement already applied: sharper prompt focused on real priorities, drift, and tradeoffs.

### Monthly and yearly

11. `1st of month 09:30` — `Lifestyle monthly review`
    - User-facing monthly review and adjustment pass.

12. `January 1st 10:00` — `Lifestyle yearly review`
    - User-facing yearly direction review.

## Important Clarifications

### The difference between `07:15` and `07:45`

- `07:15` is the conversational planning pass.
- `07:45` is the silent execution pass.
- They are intentionally separate so the user first sees the day plan, and only then does the system auto-place safe hard-time items.

### Saturday coverage

- The daily crons already run on Saturday because their cron expressions are daily, not weekday-only.
- Saturday therefore has both:
  - the normal daily morning/evening flows
  - the extra weekly review/sweep flows

### Why the poll watcher is separate

- Telegram poll sending and Telegram poll result application are not the same event.
- The watcher exists so the system can close polls after first interaction and apply results later without blocking the send cron.
- The same separation now exists for generic wrapper polls: sending the poll and replying with the captured selection are decoupled, and the minute watcher handles the follow-up.

## Improvements Already Implemented In Repo

- End-of-day Telegram poll flow is source-controlled.
- Telegram `poll_answer` runtime capture is patchable from the repo.
- Poll state is persisted under `history/polls`.
- Poll processing now supports `--no-apply` preview mode for safe smoke verification.
- Dedicated smoke scripts exist for:
  - morning
  - evening
  - weekly
  - monthly
  - yearly
  - eod poll build
  - eod poll send dry-run
  - eod poll process preview
- Morning and weekly prompts now explicitly prefer OpenViking timing memory over generic fallback slots.
- Weekly overview prompt now asks only for genuinely missing future sessions.

## Current Operational Risk

The main non-prompt operational issue still visible in live cron state is mirror write access:

- failing path: `/data/.openclaw/notion-mirror/current/data-sources`
- symptom: sync-dependent cron runs can fail or partially succeed
- current live error text already points to ownership as the fix

This is not a logic bug in the poll flow, but it is a real runtime dependency that can degrade daily scheduling and reconciliation.
